# Idea Note - 프로젝트 가이드

## 프로젝트 개요

이 저장소는 **살아가며 배웠지만 자꾸 잊어버리는 것들**을 모아두는 개인 지식 저장소입니다.

### 목적
- 책에서 읽고 고개를 끄덕였지만 일주일 뒤 희미해진 문장들
- 실수하고 나서야 깨달은 교훈들
- 누군가에게 들은, 오래 기억하고 싶은 조언
- 경험으로 얻은 나만의 원칙들

을 기록하고 오래 간직하는 것이 목표입니다.

## 디렉토리 구조

```
idea_note/
├── README.md                    # 프로젝트 소개
├── .claude/                     # Claude 설정
│   └── CLAUDE.md                # 이 파일
└── topics/                      # 주제별 노트
    ├── growth/                  # 성장, 자기계발
    │   └── thought-and-speech.md
    └── productivity/            # 생산성, 효율성
        └── effective-interview.md
```

## 카테고리 가이드

### 현재 카테고리

- **growth/**: 개인의 성장, 사고방식, 마인드셋, 자기계발 관련
- **productivity/**: 업무 효율, 시간 관리, 생산성 향상 관련

### 새로운 카테고리 제안

필요에 따라 다음과 같은 카테고리를 추가할 수 있습니다:
- **career/**: 커리어, 직업 관련 조언
- **communication/**: 의사소통, 대인관계
- **leadership/**: 리더십, 팀 관리
- **learning/**: 학습 방법, 공부 전략
- **life/**: 삶의 지혜, 일상
- **tech/**: 기술 관련 인사이트 (코딩 원칙은 아니지만 기억하고 싶은 기술적 교훈)

## 작성 가이드라인

### 파일명 규칙
- 소문자와 하이픈 사용: `thought-and-speech.md`
- 내용을 함축하는 명확한 이름
- 한글 파일명 가능하지만 영문 권장

### 내용 작성 스타일
1. **간결함**: 핵심만 담기
2. **개인적**: 나에게 의미 있는 내용
3. **실용적**: 실제 적용 가능한 조언
4. **진정성**: 경험에서 우러난 교훈
5. **유연성**: 글 형식, bullet point, 혼합 형식 모두 가능

### 마크다운 구조 예시

**스타일 1: Bullet Point 형식** (간결한 정리에 적합)
```markdown
# 제목

## 배경
왜 이것을 기록하게 되었는지

## 핵심 내용
- 주요 포인트 1
- 주요 포인트 2
- 주요 포인트 3

## 적용 방법
- 실천 방법 1
- 실천 방법 2

## 출처 (선택)
책, 사람, 경험 등
```

**스타일 2: 글 형식** (깊이 있는 성찰에 적합)
```markdown
# 제목

## 배경
어떤 상황에서 이 교훈을 얻게 되었는지 이야기 형식으로 서술.

## 핵심 내용
중요한 깨달음과 인사이트를 자연스러운 문장으로 풀어서 작성.
여러 단락으로 나눠서 작성 가능.

## 적용 방법
앞으로 어떻게 실천할 것인지에 대한 구체적인 계획.

## 출처 (선택)
책, 사람, 경험 등
```

**스타일 3: 혼합 형식** (상황에 따라 자유롭게)
```markdown
# 제목

배경 설명을 간단한 문장으로 시작.

## 핵심 내용
- 포인트 1: 구체적인 설명을 추가
- 포인트 2: 경험을 글로 풀어서 작성
- 포인트 3

중요한 부분은 단락으로 강조할 수도 있음.

## 결론
마무리 생각
```

## Claude의 역할

Claude가 이 프로젝트에서 도울 수 있는 일:

1. **새로운 노트 작성**: 사용자가 배운 교훈을 정리하여 마크다운 파일로 작성
2. **카테고리 제안**: 새로운 내용에 적합한 카테고리 추천
3. **내용 정리**: 산발적인 생각을 구조화된 노트로 변환
4. **검색 및 참조**: 기존 노트에서 관련 내용 찾기
5. **연결 제안**: 관련된 주제의 노트들을 연결

## 작업 시 주의사항

- 이 저장소는 **개인적인 성찰과 교훈**을 담는 곳입니다
- 기술 문서나 튜토리얼이 아닙니다
- 완벽한 문장보다는 **진솔한 기록**이 중요합니다
- 너무 형식적이지 않고 자연스러운 톤을 유지합니다

## 예시

좋은 노트 예시:
- "회의에서 먼저 듣고 나중에 말하기"
- "코드 리뷰는 비판이 아니라 성장의 기회"
- "완벽한 계획보다 빠른 실행이 중요한 순간"

피해야 할 내용:
- 단순한 기술 문서나 API 레퍼런스
- 다른 곳에서 쉽게 찾을 수 있는 일반적인 정보
- 개인적 의미가 없는 복붙된 내용
